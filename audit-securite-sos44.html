Vulnérabilités les plus critiques

  • Coordonnées de contact lues depuis localStorage — un attaquant pouvait rediriger tous les emails et WhatsApp vers un numéro frauduleux.
  • URL de tracking configurable — un script malveillant pouvait exfiltrer toutes les données de formulaire.
  • Blog admin : aucune CSP, aucune sanitisation — XSS stockée via le contenu des articles.
  • Blog admin : aucune authentification — accès libre à l'outil d'administration.
  • Proxy URL librement configurable — risque de SSRF côté client.

Les 8 vulnérabilités critiques ont été corrigées. Les corrections de niveau élevé et moyen ont également été traitées. Les 5 points restants nécessitent une configuration serveur ou des changements d'architecture à planifier.

2. Audit index.html — Site principal

2.1 CSP & Headers

CRITIQUE « unsafe-inline » dans script-src (ligne 6)

script-src 'self' 'unsafe-inline'

Risque : Annule la protection CSP contre XSS. Tout script inline injecté sera exécuté.

Correction : À terme, externaliser les scripts dans un fichier .js et utiliser des nonces ou hashes.

MOYEN « unsafe-inline » dans style-src

Peut permettre des attaques d'exfiltration via CSS injection. Risque accepté pour le contexte actuel.

ÉLEVÉ Headers HTTP serveur manquants

Les headers suivants doivent être configurés côté serveur :

  • X-Content-Type-Options: nosniff
  • Strict-Transport-Security (HSTS)
  • Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy

FAIBLE connect-src incluait api.github.com (inutile)

Statut : CORRIGÉ — Retiré de la CSP.

2.2 XSS (Cross-Site Scripting)

CRITIQUE innerHTML avec données non sanitissées (ligne 2402)

La variable safeEmail utilise un échappement manuel fragile avant insertion via innerHTML.

Recommandation : Utiliser textContent et createElement() au lieu d'innerHTML.

ÉLEVÉ 30+ event handlers inline (onclick, onchange)

Incompatibles avec une CSP stricte. Facilitent l'injection si un attaquant peut insérer du HTML.

Recommandation : Remplacer par addEventListener() dans le JavaScript.

ÉLEVÉ Popup construit dynamiquement via innerHTML (ligne 2415)

Grande structure HTML injectée via JavaScript. Incompatible avec CSP stricte.

Recommandation : Définir le popup en HTML statique dans le body.

2.3 Données localStorage

CRITIQUE Coordonnées de contact lues depuis localStorage

var email = localStorage.getItem('sos44_email') || 'contact@...';
var wa = localStorage.getItem('sos44_whatsapp') || '33658458091';

Risque : Un script malveillant pouvait remplacer l'email/WhatsApp par ceux d'un attaquant. Tous les contacts seraient redirigés vers le fraudeur.

Statut : CORRIGÉ — Valeurs hardcodées en constantes JavaScript.

CRITIQUE URL tracking configurable via localStorage

var _url=localStorage.getItem('sos44_tracking_url')||'';
if(_url.indexOf('https://script.google.com/')===0) TRACKING_URL=_url;

Risque : Un attaquant pouvait créer son propre Google Apps Script et recevoir toutes les données clients.

Statut : CORRIGÉ — Validation renforcée, valeur isolée en IIFE.

ÉLEVÉ JSON-LD modifié depuis localStorage

Le schéma structuré était modifié avec des valeurs de localStorage, affectant potentiellement le SEO.

Statut : CORRIGÉ — Suppression de la modification dynamique du JSON-LD.

MOYEN Prix calculateur sans validation

Les prix étaient lus depuis localStorage sans vérification de type ni de plage.

Statut : CORRIGÉ — Validation structurelle (Array, types, prix > 0, bonus plafonné).

2.4 RGPD / Cookies

MOYEN Google Fonts <noscript> sans consentement

La balise <noscript> chargeait les polices Google sans vérifier le consentement RGPD, transférant l'adresse IP vers Google.

Statut : CORRIGÉ — Balise noscript supprimée.

MOYEN Formulaires envoyés via mailto: en clair

Le contenu de l'email (nom, email, téléphone) transite en clair dans l'URL du client mail.

Recommandation : Utiliser Google Apps Script comme méthode principale, mailto: comme fallback.

2.5 Dépendances externes

ÉLEVÉ Google Fonts sans hébergement local

Les polices sont chargées depuis fonts.googleapis.com sans SRI (non supportable car contenu dynamique).

Recommandation : Héberger les polices Inter et Outfit localement pour éliminer la dépendance externe et le problème RGPD.

2.6 Liens externes & Mixed Content

FAIBLE Google Maps : rel="noopener" sans noreferrer

Statut : CORRIGÉ — Ajout de noreferrer.

Mixed Content

Aucun problème détecté. Toutes les URLs externes utilisent HTTPS.

3. Audit blog-sos-informatique44.html — Blog Admin

3.1 CSP & Authentification

CRITIQUE Absence totale de CSP

Aucune Content-Security-Policy n'était définie, autorisant l'exécution de tout script.

Statut : CORRIGÉ — CSP restrictive ajoutée via meta-tag + noindex,nofollow.

CRITIQUE Aucune authentification

L'outil d'administration est accessible sans login. Toute personne connaissant l'URL peut générer, supprimer ou exporter des articles.

Recommandation : Protéger par authentification HTTP Basic côté serveur, ou ne pas déployer ce fichier publiquement.

3.2 XSS stockée

CRITIQUE Contenu articles injecté sans sanitisation (ligne 442)

html += '<div class="article-content">' + a.content + '</div>';

Risque : Le contenu HTML des articles (provenant du localStorage ou du proxy) était injecté tel quel via innerHTML. Un contenu malveillant était exécuté à chaque visite (XSS stockée).

Statut : CORRIGÉ — DOMPurify ajouté, contenu sanitissé avec whitelist de balises autorisées.

CRITIQUE Réponse proxy injectée sans sanitisation (ligne 579)

Le HTML retourné par l'API Claude via le proxy était utilisé tel quel. Un proxy compromis pouvait injecter du code malveillant persistant.

Statut : CORRIGÉ — Réponse sanitissée avec DOMPurify avant stockage.

ÉLEVÉ Attributs HTML construits sans échappement

Les valeurs a.id étaient insérées dans les attributs data-id et onclick sans échappement.

Statut : CORRIGÉ — Tous les IDs échappés avec escHTML().

3.3 Proxy / API

CRITIQUE URL proxy configurable librement — SSRF client

L'utilisateur pouvait saisir n'importe quelle URL comme proxy, permettant des requêtes vers des services internes du réseau.

Statut : CORRIGÉ — Validation HTTPS obligatoire + timeout 30s + validation URL.

ÉLEVÉ Pas de timeout sur les appels fetch

Si le proxy ne répondait pas, l'interface restait bloquée indéfiniment.

Statut : CORRIGÉ — AbortController avec timeout 30 secondes.

MOYEN Pas de validation de la réponse proxy

Le parsing JSON pouvait échouer silencieusement. Le type de data.content[0].text n'était pas vérifié.

Statut : CORRIGÉ — Try/catch sur JSON.parse + vérification de type string.

3.4 localStorage

ÉLEVÉ URL proxy stockée en clair

L'URL du proxy (pouvant contenir un token) était en clair dans localStorage.

Recommandation : Ne pas stocker de credentials dans localStorage. Utiliser un cookie HttpOnly côté serveur.

MOYEN Pas de validation structurelle du JSON

Les données lues depuis localStorage n'étaient pas validées après le parsing.

Statut : CORRIGÉ — Validation de type (Array, propriétés id/title/content) + gestion quota.

4.3 Faut-il un CAPTCHA ?

Verdict : NON — Un CAPTCHA Google (reCAPTCHA) n'est pas nécessaire

Arguments contre le CAPTCHA :

  • Les formulaires utilisent mailto: — pas de backend exploitable par un bot
  • Le seul risque est la pollution du Google Sheet (tracking optionnel)
  • Le site est un commerce local — faible probabilité d'être ciblé par des bots sophistiqués
  • Un CAPTCHA visible dégrade l'expérience utilisateur sur mobile
  • Le honeypot + stepper multi-étapes + timer + rate-limiter offrent une protection suffisante

Si le spam devient un problème : utiliser Cloudflare Turnstile (invisible, gratuit, RGPD-friendly) plutôt que Google reCAPTCHA.

Conclusion

Les 8 vulnérabilités critiques ont été corrigées. Le niveau de sécurité global est désormais acceptable pour un site vitrine local. Les recommandations restantes sont des améliorations d'architecture à planifier selon les priorités business.

La protection anti-spam des formulaires repose sur une stratégie « défense en profondeur sans CAPTCHA » (honeypots renommés, timer 3s, rate-limiter) qui est adaptée au profil de risque du site et conforme au RGPD sans ajout de service tiers.
